The origin story

How we built a language that AI can't lie in

From a single annoyed thought to a shipped programming language with its own built-in AI — in a handful of days, for about the price of a sandwich in cloud GPU. Here's the whole messy, fun, honest story.

LambdaQ Labs · a build log

It started with a small, specific irritation. You ask an AI to write some code. It answers instantly, confidently — and calls a function that does not exist. It made it up. The code looks perfect right up until it explodes. Every developer using AI has hit this. The industry's answer is always the same: write better prompts, beg the model harder.

We didn't buy it. You can't prompt your way out of a model's imagination. So we asked a stubborn question: what if the language itself made lying impossible?

Chapter 1The bet

The idea: a programming language where an AI can't reference something that isn't real, can't quietly do things it didn't declare, and can't hand you code that fails its own tests — because the language checks all of it before you ever see the answer.

Building a compiler from scratch takes years. So we did the sensible thing and stood on giants' shoulders — we forked Roc, a beautiful existing language whose design happens to make "what can this code touch?" a checkable question. We named ours Achuk (Roc is a mythical giant bird; a achuk is what does the gripping). Then we got to work on the part nobody had done.

Chapter 2The four locked doors

Instead of one clever trick, we built four independent checks — a code has to pass all of them, and no single one is trusted:

1. A live map of what exists. The code lives in a searchable database. The AI asks "what's actually here that fits?" instead of guessing.

2. A straitjacket for the AI's pen. When the AI writes, it's physically constrained so that made-up names cannot be typed — not discouraged, impossible.

3. The real compiler, checking every line. Whatever the AI produces gets compiled for real before it's shown.

4. Tests that actually run. Code can look right and still compute the wrong answer. So the answers get executed and checked.

Then came the moment of truth. We fed a small AI a batch of tasks. Without the code map, it invented 38 fake functions across ten tasks. We turned the map on and ran it again:

# invented functions per run
before:  38
after:   0
Zero. From context alone. That was the day we knew this wasn't just a nice idea.

Chapter 3Teaching a language with no students

Here's a problem nobody warns you about: a brand-new language has no code in the world, so AI models are worst at exactly the thing they'd be used for. Every prior "AI-first language" died right here.

Our escape was to manufacture the lessons. We wrote a generator that produces thousands of correct little Achuk programs — guaranteed correct, because they're built from the rules rather than guessed. Then we rented a cloud GPU and taught a tiny AI model (small enough to run on any laptop, no internet, no API key) using them.

The first attempt taught the model to reference things by an unguessable secret code instead of by name. Useless. We caught it, fixed it, and the total cost of that training run was:

$ 0.03

Three cents. That number stops being a joke in a minute.

Chapter 4The scoreboard that mattered

We kept growing the lessons and re-teaching. The model's "makes stuff up" rate walked down a staircase: 28% clean → 89% → 100%. Then we did the honest test — we wrote fresh tasks by hand, worded differently so the model couldn't just pattern-match, and it still scored well.

Then the big one. The whole project's reason to exist rests on a single claim: can a purpose-built language beat the languages the AI already knows cold? So we had one model write the same problems in five languages, ran every single answer, and counted what actually worked:

languagehow often the code worked
Achuk (after 3¢ of training)94%
JavaScript (as-is)89%
Python (as-is)56%
Rust (as-is)35%
Go (as-is)7%

The same AI wrote 94%-correct Achuk and 56%-correct Python. We reran the whole thing with a model fourteen times bigger to be sure it wasn't a fluke of small models — Achuk still came first. That's the thesis, on the board, with the scoreboard attached and the tools to reproduce it published.

Chapter 5The part where it becomes real

A proof isn't a product. So we built the boring, essential everything: a one-command installer, a self-updater, editor support, a package registry with a rule no other registry has (every package must be readable by AI, or it's rejected), a playground that runs the real engine in your browser with zero servers, and connectors so Claude, Cursor, and friends plug straight in.

We put the whole thing on our own hardware. Monthly cost to run the website, the registry, and the data pipeline:

$ ~0.00

Then we did the thing every honest builder should and almost nobody does: we used our own product from scratch. Fresh install, build a real package. And it immediately caught a bug — installing a package was quietly forgetting to tell the AI about it. We'd have shipped that. Dogfooding earned its keep in the first ten minutes.

Chapter 6One download, batteries included

Here's the line we cared about most. You said it plainly: a language whose whole point is the AI, that makes you download the AI separately, isn't a product — it's a chore. Dead right.

So we made it one file. The installer brings the compiler, the editor tools, the AI model, and the engine that runs it — everything, in a single command. Type this on a fresh machine and, seconds later, an AI on your own laptop writes verified code with no key and no cloud:

$ curl -fsSL https://achuk.dev/install.sh | sh
$ achuk ai gen "define atleast10 that takes the bigger of x and 10"
── generated ──  atleast10 = |p0| Nat.max(p0, 10)
── verified ── real compiler: OK

And when the little model gets it wrong — which it does — you don't get a bug. You get this, and the broken code never reaches you:

── REJECTED ── the compiler found an error

Chapter 7The hunt for a better brain (we're in it right now)

The bundled model is deliberately tiny — it has to run on any laptop. It's reliable but not brilliant, and it fumbles unusual problems (the guardrails catch those, which is the point, but still). So the current chapter, being written as you read this, is the search for a bigger-but-still-tiny brain. We tested Google's Gemma, and it scored higher — but it "thinks out loud" before answering, which fights our straitjacket and is slow. We tried a couple of Qwen models and hit a bug in our own training shortcut that made a great model look terrible; we caught it by noticing the damage was identical no matter how much we trained. Now we're re-running the whole comparison with the proven tools.

That's the honest state of it: no clean winner yet, but we know exactly why the last test was messy, and the fix is running. Total spent chasing this: under a dollar.
